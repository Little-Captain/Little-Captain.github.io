1. shell 编程基础

1.1. 什么是 shell 脚本

  • 最简单的解释是, shell 脚本是一个包含一系列命令的文件。shell 读取这个文件, 然后执行这些命令, 就好像这些命令是直接输入到命令行中一样。

  • shell 很独特, 因为它既是一个强大的命令行接口, 也是一个脚本语言解释器。

1.2. 怎样写 shell 脚本

1.2.1. 基本步骤

  • 第一步：编写脚本

    • shell 脚本是普通的文本文件。所以需要一个文本编辑器来编辑它。
    • 脚本的第一行一般以 "#!" 开头, 它看起来像是注释, 但是它应该具有一定的意义, 所以它不仅仅是注释。
    • 实际上, 这个 "#!"字符序列是一种特殊的结构, 称之为 shebang。shebang 用来告知操作系统, 执行后面的脚本应该使用的解释器的名字。每一个 shell 脚本都应该将其作为第一行。

  • 第二步：使脚本可执行

    • 系统相当严格, 它不会将任何老式(没有执行权限)的文本文件当做程序。所以需要将脚本文件的权限设置为允许执行。
    • 对于脚本, 有两种常见的权限设置: 权限为 755 的脚本, 每个人都可以执行; 而权限为 700 的脚本, 则只有脚本所有人才能执行。
    • 注意, 为了能够执行脚本, 它必须是可读的。

  • 第三步：将脚本放置在 shell 能够发现的位置

    • 当没有显式指定路径名时, shell 会自动地寻找 PATH 环境变量包含的目录, 来查找可执行文件。
    • 脚本的理想位置
      • 通常 ~/bin 目录是一个存放个人使用脚本的理想位置。
      • 如果编写了一个系统上所有用户都可以使用的脚本, 则该脚本的传统位置是 /usr/local/bin。
      • 系统管理员使用的脚本通常放置在 /usr/local/sbin。
      • 在大多数情况下, 本地支持的软件, 无论是脚本或者是编译好的程序, 都应该放置在 /usr/local 目录下, 而不是 /bin 或是 /usr/bin 目录下。这些目录都是由 Linux 文件系统层次结构标准指定的, 只能包含由 Linxu 发行商所提供和维护的文件。

1.2.2. 编写 shell 脚本的一些技巧

  • 长选项名

    • 为了减少输入, 当在命令行中输入选项时, 短选项更可取。但是, 在编写脚本时, 长选项名可以提高可读性。

  • 缩进和行连接

    • 当使用长选项命令时, 将命令扩展为好几行, 可以提高命令的可读性。
    • 通过行连接符(反斜杠-回车符序列)和缩进, 可以让一个复杂的命令更容易理解。该技术在命令行中也奏效, 但是很少使用, 原因就是在输入和编辑时会相当麻烦。
    • 脚本和命令行的一个区别是, 脚本可以使用制表符来实现缩进, 但在命令行中, Tab键用来激活自动补齐功能。

  • 一个带引号的字符串可以包含换行符, 因此也就可以包含多个文本行。shell 将持续读取文本, 直到读取到下一个引号为止。它在命令行中也是这样工作的。

  • 示例

echo \
"<HTML>
    <HEAD>
        <TITLE>Page Title</TITLE>
    </HEAD>
    <BODY>
        Page body.
    </BODY>
</HTML>"
  • 保持脚本可运行
    • 在开发程序时, 让程序保持可运行的状态会非常有用。
    • 通过这种方式, 并经常测试, 就可以在开发过程的早期检测到错误。这也会让问题的调试变得容易。
    • 通过添加空函数(程序员将其称为 stub), 可以在早期验证程序的逻辑流程。当构建一个 stub 时, 最好是包含一些能够为程序员提供反馈信息的东西, 这些反馈信息可以显示正在执行的逻辑流程。

1.3. here 文档

  • 有一个称之为"hure 文档"或"here 脚本" 的方法来输出文本。here 文档是 I/O 重定向的另外一种形式, 在脚本中嵌入正文文本, 然后将其输入到一个命令的标准输入中。

  • 工作方式

# command 是接受标准输入的命令名, token 是用来指示偏入文本结尾的字符串。
# 注意，token 必须在一行中单独出现，而且文本行的末尾没有空格。
command << token
text
token
  • 示例
cat << _EOF_
<HTML>
    <HEAD>
        <TITLE>${TITLE}</TITLE>
    </HEAD>
    <BODY>
        <H1>${TITLE}</H1>
        <P>${TIME_STAMP}</P>
    </BODY>
</HTML>
_EOF_

  • 在默认情况下, here 文档内的单引号和双引号将失去它们在 shell 中的特殊含义。也就是说可以在 here 文档中随意嵌入引号。

  • here 文档可以和能够接受标准输入的任何命令一起使用。

  • 如果将重定向操作符由 "<<" 改为 "<<-", shell 就会忽略在 here 文档中行开头的 Tab 字符。这样就能缩进 here 文档, 从而提供可读性。

1.4. shell 脚本中的变量和常量

1.4.1. 创建变量和常量

  • 当 shell 遇到一个变量时, 会自动创建这个变量。这点与大多数程序中, 使用一个变量时必须先声明或定义有所不同。

  • 在这个方面, shell 非常的宽松, 但也会导致一些问题。

  • 一个不存在的变量被参数扩展后为空。

foo='yes'
echo $foo # yes
echo $foo1 #

  • 变量的命名规则

    • 变量名称应由字母、数字和下划线组成。
    • 变量名称的第一个字符必须是字母或者下划线。
    • 变量名称中不允许空格和标点。

  • 变量意味着值会发生变化, 并且在大多数应用中, 变量就是这样使用的。

  • 常量是一个有名称和确定值的变量。区别就是, 常量的值不会发生变化。

  • 对于 shell 来说, 变量和常量没有区别, 这种划分更多的是为了编程者的方便。较为普道的约定是, 使用大写字母表示常量, 使用小写字母表示变量。

  • 实际上, shell 还提供了强制常量不发生变化的方法, 这是通过使用带有 -r 选项(只读)的 delcare 内置命令实现的。这个特性很少使用。

# shell 将阻止一切后续的向 TITLE 的赋值。
declare -r TITLE="Page Title"

1.4.2. 为变量和常量赋值

  • 变量赋值方式
# 其中, variable 是变量的名称, value 是变量的值。
variable=value

  • 和大多数的编程语言不同, shell 并不关心赋给变量的值的类型, 它会将值都当成字符串。通过使用带 -i 选项的 declare 命令, 可以强制 shell 将变量限制为整型数值。这如同将变量设置成只读模式一样, 实践中很少这样做。

  • 注意, 在赋值时, 变量名、等号和值之间不能含有空格。变量的值中可以包含可以扩展为字符串的任意值。

  • 示例

a=z b="a string" # 可以在一行中给多个变量赋值
c="a string and $b"
d=$(ls -l foo.txt)
e=$((5 * 7))
f="\t\ta string\n"
  • 在扩展期间, 变量名称可以用花括号 "{}" 括起来。当变量名因为周围的上下文而变得不明确时, 这就会很有帮助。

1.5. shell 函数

  • shell 函数是位于其他脚本中的迷你脚本, 可以用作自主程序(autonomous program)。

  • shell 函数有两种语法形式。

    • 这两种形式是等价的，并且可以交替使用。
# 第一种
function name {
    commands
    return
}
# 其中 name 是指这个函数的名称, commands 是这个函数中的一系列命令。

# 第二种
name() {
    commands
    return
}
# 省略了 function，但是 () 不能省略。

  • return 命令终止函数的执行, 然后将控制权还给函数调用后面的代码。

  • 注意, 为了让函数调用被识别为 shell 函数, 而不是被解释为外部程序的名字, shell 函数的定义义在脚本中的位置必须在它被调用的前面。

  • shell 函数的命名规则和变量相同。

  • 一个函数必须至少包含一条命令。return 命令(可选的)可以满足该要求。

  • shell 函数可以很好地取代别名, 并且实际上也是创建个人使用的小命令的首选方法、别名非常局限于命令的种类和它们支持的 shell 特性, 而 shell 特性则允许任何可以编写为脚本的东西。

1.6. 局部变量

  • 全局变量在整个程序期间会一直存在。在很多情况下, 这都不错。但是有时候,它会让 shell 函数的使用变得复杂。

  • 在 shell 函数中, 经常需要的是局部变量。局部变量仅仅在定义它的 shell 函数中有效, 一旦 shell 函数终止, 它就不再存在了。

  • 局部变量可以让程序员使用已经存在的变量名称, 无论是脚本中的全局变量, 还是其他 shell 函数中的变量, 而不用考虚潜在的命名冲突。

  • 局部变量是通过在变量名前面添加单词 local 来定义的。这样, 就创建并同时定义了一个 shell 函数中的局部变量。一旦出了这个 shell 函数, 这个局部变量将不再存在。

  • 这个特性让开发者编写的 shell 函数相互独立, 而且也独立于它们所在的脚本。这非常有用, 因为它有助于阻止程序中各个部分的相互干扰。该特性也可以让开发者编写出可移植的 shell 函数。

  • 也就是说, 开发者可以根据需要将某一个脚本中的 shell 函数剪切下来, 然后粘贴到另外一个脚本中。

  • 示例

function_name () {
    local foo
    foo=1
    echo "foo = ${foo}"
}

1.7. 面向过程变成之自顶向下设计

  • 在设计任何大型项目时, 最好将庞大的、复杂的任务, 拆分成一系列小的、简单的任务。

  • 先确定上层步骤, 然后再逐步细化这些步骤的过程, 称为自顶向下设计。

  • 通过这种设计方式, 可以将大而复杂的任务分解成很多小而简单的任务。

  • 自顶向下设计是一种设计程序的常见方式, 尤其适合 shell 编程。
