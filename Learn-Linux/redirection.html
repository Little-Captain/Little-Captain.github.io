1. 重定向

1.1. I/O 重定向

  • 可以把命令行的输入重定向为从文件中获取内容

  • 可以把命令行的输出结果重定向到文件中

  • I/O 重定向可以改变输出内容发送的目的地，也可以改变输入内容的来源地

    • 通常来说，输出内容显示在屏幕上，输入内容来自于键盘

1.2. 标准输入、标准输出和标准错误

  • 标准输出(standard output)

    • stdout

  • 标准错误(standard error)

    • stderr

  • 标准输入(standard input)

    • stdin

  • 默认情况下，标准输出和标准错误都被输出到屏幕上，不会保存再磁盘中

  • 默认情况下，标准输入连接到键盘

1.3. 标准输出重定向

  • 处理不想要的输出

    • 通过把不想要的输出重定向到特殊文件 /dev/null 来实现

  • 注意

    • 重定向时，只有 command、file 左右可以加空格
    • 注意这些格式：2>、2>&1、&> file

>

  • 重定向标准输出(stdout)内容的目的地

  • 每次重定向，目的文件都会被新内容重新覆盖

command > file

1.3.1. >>

  • 从文件的尾部开始添加标准输出内容

  • 如果目的文件不存在，会创建目的文件

command >> file

1.4. 标准错误重定向

  • 一个程序可以把生成的输出内容发送到任意文件流中

  • shell 使用文件描述符索引文件流，前 3 个分别为

    • 0 标准输入文件
    • 1 标准输出文件
    • 2 标准错误文件

  • shell 提供了使用文件描述符编号来表示重定向文件的方法

  • 2> 和 2>>

    • 注意没有空格

1.4.1. 将标准输出和标准错误重定向到同一个文件

  • 方法一
command > file 2>&1
# 首先重定向标准输出到 file 中
# 然后使用 2>&1 把标准错误重定向到标准输出中
# 执行两个重定向，执行效率低，重定向顺序不能换
  • 方法二
command &> file
# &> 会同时把标准输出和标准错误都重定向到 file 中
# 效率比方法一更高

1.5. 标准输入重定向

  • 使用 <
command < file # 重定向标准输入(stdin)的源
  • 示例
cat < file

1.6. 管道

  • 使用 |
# 把一个命令的标准输出传送到另一个命令的标准输入中
command1 | command2
  • 示例
ls -l /usr/bin | less
  • shell 处理管道的方式
    • 在 shell 中，管道会创建子 shell。
    • 类 UNIX 系统的子 shell 会复制所需的 shell 环境。进程结束后，所复制的 shell 环境即被销毁。
    • 这意味着子 shell 永远不会改变父 shell 的环境。

1.7. 过滤器

  • 过滤器接受输入，按照某种方式对输入进行改变，然后再输出它

  • 相关过滤器命令(参考)

    • sort：排序
    • uniq： 报告或忽略文件中重复的行

1.8. 命名管道

  • 大多数类 UNIX 系统都支持创建一个叫做命名管道的特殊类型的文件。

  • 使用命名管道可以建立两个进程之间的通信, 并且可以像其他类型的文件一样使用。

  • 客户端/服务器模式是一种常见的程序设计结构。它可以使用命名管道这样的通信方式, 也可以使用网络连接这样的进程间通信方式。

  • 命名管道的工作方式与文件相似, 但实际上是两块先进先出(FIFO)的缓冲区。与普通的管道一样, 数据从一端进入, 从另一端出来。

  • 命名管道的使用方式

process1 > named_pipe
# 以及
process2 < named_pipe
# 执行效果等价于
process1 | process2

1.8.1. 设置命名管道

  • 首先, 必须创建一个命名管道。使用 mkfifo 命令即可完成。
mkfifo pipe1
ls -l pipe1
# 这里使用 mkfifo 命令创建一个名为 pipe1 的命名管道。
# 使用 ls 命令查看文件属性, 可以看到属性字段的第一个字母是 p,
# 这表明它是一个命名管道。

1.8.2. 使用命名管道

  • 为了说明命名管道是如何工作的, 需要两个终端窗口(或者两个虚拟的控制台)。

  • 在第一个终端中, 输入一条简单的命令, 并将其输出重新定位到这个命名管道。

ls -l > pipe1
# 按下 Enter 键之后, 这个命令看起来像挂起来了。
# 这是因为从管道的另一端还没有接收到数据。
# 当这种情况发生时, 也就是说命名管道被阻塞。
# 一旦将一个进程连接到管道的另一端时, 进程会从管道中读取数据。
  • 使用第二种终端窗口, 输入连接管道的命令。第二个终端必须切换到和第一个终端相同的目录才能成功连接。
cat < pipe1
# 第一个终端窗口产生的目录列表作为 cat 命令的输出出现在第二个终端窗口。
# 一旦它不处于阻塞状态, 第一个终端窗口中的 ls 命令就成功完成了。
